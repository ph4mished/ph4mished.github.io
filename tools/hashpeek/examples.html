üîß Advanced Usage Scenarios

üìä Forensic Analysis - Memory Dumps

Extract potential hashes from binary memory dumps

./hashpeek -f memory.dmp -pb 32 -me 3.8 -ovh -t 4

Explanation:
- -pb 32: Probe for 32-byte sequences
- -me 3.8: Minimum entropy 3.8 to filter random data
- -ovh: Only output valid hashes
- -t 4: Use 4 threads for faster processing

üîç Log File Analysis

Extract hashes with context from application logs

./hashpeek -f app.log -e-ctext 'token,password,secret' -me 3.4 --json

Explanation:
- -e-ctext: Extract context around keywords
- --json: Output in JSON format for further processing

Extract specific fields from structured files

# From /etc/shadow files ./hashpeek -f shadow.txt -fd '{2} :' -j # From CSV files ./hashpeek -f users.csv -fd '{3} ,' -hc # From colon-delimited data ./hashpeek -f data.txt -fd '{2} :' -ni

‚ö° High-Performance Bulk Processing

Process large files efficiently

./hashpeek -f huge_dump.txt -mr 1000 -t 8 -ml 1024 --json > results.json

Explanation:
- -mr 1000: Limit to 1000 results
- -t 8: Use 8 threads
- -ml 1024: Limit to 1GB RAM
- --json: Output to file for analysis

üí° Pro Tips

Optimizing Performance

  • Use -t with your CPU core count for large files
  • Set -ml to prevent memory exhaustion on huge files
  • Use -mr to limit results when you only need samples

Improving Accuracy

  • Start with -me 3.5 and adjust based on results
  • Use -ovh when you only want valid hashes
  • Combine -pb with entropy filtering for binary files

Workflow Integration

  • Pipe results to other tools: hashpeek -f file.txt | grep MD5
  • Use JSON output for automated processing
  • Combine with grep/sed for complex extraction pipelines
